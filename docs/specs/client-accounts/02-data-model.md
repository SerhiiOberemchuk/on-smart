# Client Accounts — 02 Data Model

> **Status:** normative. **Depends on:** [00-overview.md](./00-overview.md), [01-auth.md](./01-auth.md). **Consumed by:** rollout Phase 2.
> Invariants #4 and #5 apply throughout: `auth-schema.ts` is CLI-generated only; migrations are generated by agents but **applied exclusively by the owner**.

## 1. Ownership boundary

- **Better Auth tables** (`user`, `session`, `account`, `verification`) live in root `auth-schema.ts`. No hand edits, no `additionalFields` — avoiding regen drift is worth more than embedding profile columns in `user`.
- **All business data** introduced by this feature lives in new files under `db/schemas/`, one domain per file, `*.schema.ts` naming, types exported via `$inferSelect` (AGENTS.md "Types — Schemas Are the Single Source of Truth").

## 2. New table: `customer_profiles` (`db/schemas/customer-profile.schema.ts`)

One row per user. Lifecycle: created/updated by the **inline mini-forms on the single-page checkout** (decision #12 — see [04-checkout-integration.md §2](./04-checkout-integration.md)) and editable manually in `/account/profilo`; created on whichever happens first. ## 5. `orders.userId` policy

- Column **already exists**: nullable `varchar(36)` with index `orders_userId_idx` (`db/schemas/orders.schema.ts` lines 31, 93). No rename, no type change.
- **Add**: FK → `user.id` with `onDelete: "set null"` (deleting a user must never delete orders — they are business records), and a Drizzle relation `orders.user` for admin joins.
- **The column stays nullable in the DB forever.** Historic guest orders keep `userId = null` permanently (binding decision #6: never backfilled).
- "Every new order has a userId" is enforced **in application code only** — `create-account-order.ts` requires a session by design, and the guest order path disappears with the wizard at Phase 8 — never as a DB constraint ([04-checkout-integration.md §3](./04-checkout-integration.md)).

## 6. Migration plan (owner-applied — invariant #5)

Phase 2 deliverable from the implementing agent:

1. Schema files from §§2–5 added under `db/schemas/` (+ the FK/relations edit in `orders.schema.ts`). `drizzle.config.ts` already includes `./db/schemas` — no config change.
2. Generated SQL via `drizzle-kit generate` → new file(s) in `drizzle/` (next number after the current `0012`). **The agent generates but never applies.**
3. A short human-readable summary for owner review: every `CREATE TABLE` / `ALTER TABLE` statement, expected order (tables before FKs; the `orders` FK requires existing `user` rows only for non-null values — historic nulls are unaffected), and rollback statements (`DROP TABLE customer_profiles / user_addresses / wishlist_items`; `ALTER TABLE orders DROP FOREIGN KEY …`).
4. **STOP.** Implementation continues only after the owner confirms the migration is applied in production (Phase 2 acceptance).

## 7. Cache-tag registry additions (used from Phase 4/5, declared here)

Per code-style-rules §6, new tag namespaces are added to `types/cache-trigers.constant.ts` **only if** account data is ever cached. **v1 does not cache any per-user reads** (invariant #3; normative rules in [03-account-area.md](./03-account-area.md)) — therefore **no new CACHE_TAGS namespaces are introduced in v1**. This is deliberate: an unused `CACHE_TAGS.account` namespace would invite someone to use it. If a future measured need arises, follow code-style-rules §17.
